Input — pick the defaults

Four visual decisions. Click into each input to feel focus state. Then reply with letters (e.g. 1B 2A 3B 4B).

DECISION 1

Default height: 32px or 36px?

Mockup proposes 32px (tighter, claude.ai-style). Button md = 36px. Matching Button gives a uniform vertical rhythm in modal forms (input + adjacent button = same height). 32px feels denser, 36px feels calmer.

A · 32px (tight, mockup)
Visible to all members
B · 36px (matches Button md)
Visible to all members
DECISION 2

Corner radius: 8px or 12px?

Button is locked at 8px. Inputs in current code mostly use 12px (rounded-xl) — auth pages especially. 8px on inputs ties the form together visually with buttons; 12px keeps the existing softer auth feel.

A · 8px (matches Button)
B · 12px (current auth pages)
DECISION 3

Error color: bark or copper?

Code currently uses bark for error helper text (warmer brown). Button destructive is locked at copper. Aligning Input errors to copper means error state matches destructive action color across the app — same semantic.

A · BARK (current code)
Slug must be lowercase alphanumeric.
B · COPPER (Button-aligned)
Slug must be lowercase alphanumeric.
DECISION 4

Hover state: none or border bumps to mist?

Most current inputs have no hover state — only focus. Mockup proposes a subtle border-color bump to mist on hover so the input visibly responds before you click. Hover over each below to feel the difference.

A · NONE (current)
B · MIST BUMP (mockup)
